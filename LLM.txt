# YAR Language Reference for LLMs

YAR is a small, compiled language with direct syntax, explicit types, errors as values, and LLVM-based native code generation via clang.

Implementation note: the shipped compiler CLI is the Rust 2024 `yar` binary from `crates/yar-cli`; `crates/yar-compiler` owns Rust token, diagnostic, AST, lexer, parser, package-graph loading, package-lowering, monomorphization, semantic checking, and LLVM emission. It checks and emits clang-accepted LLVM for every checked-in `testdata/**/main.yar` entry program, including stdlib-backed host packages and the native main wrapper. `crates/yar-cli` provides `check`, `emit-ir`, `build`, host `run`, host `test`, `init`, dependency manifest, lock, fetch, and update commands, plus origin-scoped package loading. Native build/run/test paths link the Rust runtime archive from `YAR_RUNTIME_ARCHIVE`, a sibling `libyar_runtime.a`/`yar_runtime.lib`, or the workspace `target/release` archive. `crates/yar-runtime` is the Rust runtime crate and exports the C ABI `yar_*` helpers used by generated code for allocation, strings, maps, string builders, taskgroups, channels, argv, environment, child processes, filesystem, and TCP networking.

## Project Structure

```
myproject/
  yar.toml          # optional manifest for external dependencies
  yar.lock          # auto-generated lock file (commit to VCS)
  main.yar          # entry package, must have `package main` and `fn main() i32`
  main_test.yar     # test file (excluded from normal builds)
  helpers.yar       # additional file in same package
  util/
    util.yar        # imported package, `package util`
```

## File Shape

```yar
package main

import "util"
import "strings"

struct Point {
    x i32
    y i32
}

fn main() i32 {
    p := Point{x: 1, y: 2}
    print(to_str(p.x + p.y) + "\n")
    return 0
}
```

- Entry package: `package main` with `fn main() i32` or `fn main() !i32`
- Imports: `import "path"` (slash-separated, qualified usage: `path.func()`)
- Package identity: `PackageId = (source origin, source-relative subpath)`;
  import text and dependency aliases are bindings, not identity
- Resolution is scoped to the importer origin: same-origin packages → aliases
  directly declared by that origin → embedded stdlib
- Imports inside embedded stdlib are sealed to the stdlib origin
- Distinct imports in one package cannot have the same final qualifier segment
- Visibility: `pub` on struct/fn/enum/interface to export; private by default
- Comments: `// line comments` only

## Declarations

```yar
x := 42                    // short declaration (type inferred)
var count i32 = 0          // typed declaration with initializer
var user User              // typed declaration (zero value)
```

Locals are block-scoped, reassignable, cannot be redeclared in same scope.

## Structs

```yar
struct User {
    id i32
    name str
}

// Usage
u := User{id: 1, name: "bob"}
u.name = "eve"
empty := User{}
```

No embedding, no tags. Recursive only through pointers: `next *Node`.

## Enums

```yar
enum Color { Red, Green, Blue }

enum Expr {
    Int { value i32 }
    Name { text str }
}

// Usage
c := Color.Red
e := Expr.Int{value: 42}

// Single-field cases also accept positional syntax
e2 := Expr.Int(42)            // same as Expr.Int{value: 42}
e3 := Expr.Name("main")      // same as Expr.Name{text: "main"}

// Match (exhaustive, statement-form only)
match e {
case Expr.Int(v) {
    print(to_str(v.value))
}
case Expr.Name(v) {
    print(v.text)
}
}

// Ignore payload with _
match e {
case Expr.Int(_) { print("int") }
case Expr.Name(_) { print("name") }
}
```

No `==`/`!=` on enums. No recursive enums (use pointers).

## Interfaces

```yar
interface Writer {
    write(msg str) !void
}

struct Logger { prefix str }

fn (l Logger) write(msg str) !void {
    print(l.prefix + msg)
    return
}

fn emit(w Writer) !void {
    return w.write("ok")
}

// Implicit satisfaction: Logger satisfies Writer
```

- Method requirements only (no fields)
- Implicit satisfaction (no `implements` keyword)
- Exact receiver matching: `T` and `*T` are distinct
- Not comparable, not generic
- Zero-value interface panics on method call


- Capture by value (read-only inside closure)

## Generics

```yar
struct Box[T] {
    value T
}

fn first[T](values []T) T {
    return values[0]
}

// Always explicit type arguments
b := Box[i32]{value: first[i32]([]i32{7, 9})}
```

- On top-level `struct` and `fn` only
- No inference, no constraints
- Not on enums, methods, or interfaces
- Monomorphized before checking

## Pointers

```yar
struct Node {
    value i32
    next *Node
}

n := &Node{value: 1, next: nil}
n.value = 2                 // implicit dereference for pointer-to-struct
(*n).value = 2              // explicit dereference also works
p := &n.next                // address-of field through pointer
```

- `&expr` (address-of), `*ptr` (dereference), `nil`
- `ptr.field` works for pointer-to-struct (implicit dereference for field access)
- `nil` requires pointer-typed context (`p := nil` is rejected)
- Dereferencing `nil`, explicitly or through field access, terminates with
  `runtime failure: nil pointer dereference`
- No pointer arithmetic

## Arrays and Slices

```yar
// Arrays (fixed size)
nums := [3]i32{1, 2, 3}
first := nums[0]
nums[1] = 10

// Slices (dynamic)
values := []i32{}
values = append(values, 1)
values = append(values, 2)
part := values[0:1]
tail := values[1:]           // open-ended: end defaults to len(values)
head := values[:2]           // open-ended: start defaults to 0
```

Array and slice indexing is runtime bounds-checked. `len()` works on both.
`append()` works on slices only.

## Maps

```yar
counts := map[str]i32{"a": 1}
counts["b"] = 2

// Lookup returns !V (errorable)
x := counts["a"]?                      // propagate error
val := counts["a"] or |err| { return 0 }  // handle locally

if has(counts, "a") { /* ... */ }
delete(counts, "b")
k := keys(counts)    // []str snapshot, unordered
n := len(counts)
```

Missing key returns `error.MissingKey`.

## Control Flow

```yar
// If/else
if x > 0 {
    print("positive")
} else if x == 0 {
    print("zero")
} else {
    print("negative")
}

// For (unconditional)
for {
    i += 1
    if i == 10 { break }
}

// For (condition)
for i < 10 {
    i += 1
}

// For (C-style)
for i := 0; i < 10; i += 1 {
    print(to_str(i))
}

// break, continue work in loops
```

No `while`, no range-based for, no `for..in`.

Concurrency adds:

```yar
results := taskgroup []i32 {
    spawn square(2)
    spawn square(3)
}
```

- `taskgroup []R { ... }` is an expression returning `[]R`
- `spawn call(...)` is valid only inside a taskgroup body and accepts a named
  function or an immediately called inline function literal
- `taskgroup []void` and `taskgroup []!T` are valid

## Operators

Arithmetic: `+`, `-`, `*`, `/`, `%`
`i32`/`i64` addition, subtraction, multiplication, and negation wrap to the
operand width. Division and remainder trap on zero divisors and `MIN / -1`.
Comparison: `<`, `<=`, `>`, `>=`, `==`, `!=`
Boolean: `&&` (short-circuit AND), `||` (short-circuit OR)
Unary: `-` (negate), `!` (not), `&` (address-of), `*` (dereference)
Compound assignment: `+=`, `-=`, `*=`, `/=`, `%=`. The target is evaluated
once. Map elements do not support compound assignment because lookup is
errorable.
String: `+` (concatenation), `==`/`!=` (equality), `s[i]` (byte), `s[i:j]`, `s[i:]`, `s[:j]` (slice)

## Error Handling

Errors are values. No exceptions.

```yar
// Define errorable function
fn divide(a i32, b i32) !i32 {
    if b == 0 {
        return error.DivideByZero
    }
    return a / b
}

// Propagation with ? (caller must also be errorable)
result := divide(10, 2)?

// Local handling with or
result := divide(10, 0) or |err| {
    print("error!")
    return 0    // must exit block (return/break/continue)
}

// Direct return of errorable
fn wrapper() !i32 {
    return divide(10, 2)
}

// Error comparison
fn handle() !i32 {
    result := divide(10, 0) or |err| {
        if err == error.DivideByZero {
            return -1
        }
        return error.Unknown
    }
    return result
}
```

- `?` propagates error to caller (caller must return `!T` or `error`)
- `or |err| { ... }` handles error locally; block must terminate control flow when used in value position
- `error.Name` creates named error values
- `error` is a valid type for variables, parameters, fields
- Raw errorable call expressions cannot be used directly in expressions; must use `?`, `or`, or `return`
- First-class `!T` values produced by the language, such as taskgroup result elements, may later be handled with `?` or `or`

`!void` functions use bare `return` for success:

```yar
fn save(data str) !void {
    fs.write_file("out.txt", data)?
    return
}
```

## Concurrency

```yar
jobs := chan_new[i32](4)
results := chan_new[i32](4)

taskgroup []void {
    spawn producer(jobs)
    spawn worker(jobs, results)
}
```

- `spawn` requires a named function call or an immediately called inline
  function literal; arbitrary function values, builtins, and methods cannot be
  spawned; direct host-intrinsic spawns require task-wrapper support, which is
  currently available only for `fs.read_file`
- spawned call return type must match the taskgroup element type exactly
- spawn arguments and inline-literal captures must be recursively share-safe
- share-safe inputs are scalars, `str`, `error`, arrays, enums, non-resource
  structs, `!T`, and `chan[T]` when every contained type is share-safe;
  `!void` is also share-safe
- pointers, slices, maps, interfaces, functions, resource structs, and values
  containing them cannot cross a spawn boundary
- result types are unrestricted because results become visible only after join
- `spawn` is rejected inside a function literal nested under the same taskgroup
- `return` is rejected inside a taskgroup body
- `?` is rejected inside a taskgroup body because it could bypass the join;
  nested function literals remain separate propagation scopes
- `break` / `continue` cannot jump out through an enclosing loop outside the taskgroup
- channels are bounded FIFO values
- `chan_send` / `chan_recv` report `error.Closed` on closed-channel failure
- channels support `==` and `!=` identity comparison
- each successful `spawn` starts one POSIX thread immediately; Windows
  concurrency calls fail at runtime
- arguments and captures are shallow copies, not deep copies; bare `i64`
  handles remain indistinguishable from ordinary scalar integers to the checker
- runtime-backed `i64` handles are kind-checked, non-reused process-local
  registry IDs with synchronized state; this prevents invalid pointer
  dereferences but does not add compile-time nominal types or provenance

`fs.File` is a resource struct and cannot cross a spawn boundary.

stale, and wrong-kind IDs produce `error.Closed`.

### net

Types:

```
net.Addr { host str, port i32 }
net.Conn { handle i64 }
net.Listener { handle i64 }
```

`net.Conn` and `net.Listener` are resource structs and cannot cross a spawn
boundary. Raw network handles represented as `i64` are not distinguished from
ordinary integers by the spawn checker. Runtime lookup validates ID and kind,
synchronizes per-handle state, never reuses IDs, and maps unknown, stale, or
wrong-kind IDs to `error.Closed`. Explicit close removes the ID so new lookups
fail, then waits for an operation holding the socket lock before releasing it;
close does not interrupt blocking I/O. This is runtime safety, not compile-time
nominal or provenance safety.

Listeners and connections are opaque registry-backed `i64` handles. All calls
are blocking.

```
net.listen(host str, port i32) !i64          // bind + listen; "" = all interfaces
net.accept(listener i64) !i64                // block until connection arrives
net.listener_addr(listener i64) !net.Addr    // bound address (port 0 discovery)
net.close_listener(listener i64) !void       // close listener
net.connect(host str, port i32) !i64         // TCP connect with DNS resolution
net.read(conn i64, max_bytes i32) !str       // read up to N bytes; "" on EOF
net.write(conn i64, data str) !i32           // write all data; returns bytes written
net.close(conn i64) !void                    // close connection
net.local_addr(conn i64) !net.Addr           // local endpoint
net.remote_addr(conn i64) !net.Addr          // remote endpoint
net.set_read_deadline(conn i64, millis i32) !void   // 0 = no timeout
net.set_write_deadline(conn i64, millis i32) !void  // 0 = no timeout
net.resolve(host str, port i32) !net.Addr    // DNS resolution
net.listen_stream(host str, port i32) !net.Listener
net.connect_stream(host str, port i32) !net.Conn
```

### http

Types:

```
http.Request { method str, path str, headers map[str]str, body str }
http.Response { status i32, headers map[str]str, body str }
```

Functions:

```
http.text(status i32, body str) http.Response
http.serve(addr net.Addr, handler fn(http.Request) !http.Response) !void
```

`http.serve` listens on a TCP address and processes connections sequentially.
It reads one HTTP/1.1 request per connection, calls the handler, writes one
response, and closes the connection before accepting the next one. Handler
errors become `500 Internal Server Error` responses and the server keeps
accepting. Malformed requests receive `400 Bad Request`. Header names in
`Request.headers` are lowercase. There is no keep-alive, router, query parser,
middleware, auth, TLS, or stdlib session/JWT support.

Example:

```yar
import "http"
import "net"

fn handle(req http.Request) !http.Response {
    return http.text(200, "hello\n")
}

fn main() !i32 {
    http.serve(net.Addr{host: "127.0.0.1", port: 8080}, fn(req http.Request) !http.Response {
        return handle(req)
    })?
    return 0
}
```

## Testing

### Test Files

Files ending in `_test.yar` in the same package. Excluded from `build`/`run`/`check`.

### Test Functions

```yar
package main

import "testing"

fn add(a i32, b i32) i32 {
    return a + b
}

fn test_add(t *testing.T) void {
    testing.equal[i32](t, add(2, 3), 5)
}

fn test_add_negative(t *testing.T) void {
    testing.equal[i32](t, add(-1, 1), 0)
}
```

Convention: `fn test_<name>(t *testing.T) void`

### Testing Errors

```yar
fn divide(a i32, b i32) !i32 {
    if b == 0 {
        return error.DivideByZero
    }
    return a / b
}

fn test_divide_ok(t *testing.T) void {
    result := divide(10, 2) or |err| {
        testing.fail(t, "unexpected error")
        return
    }
    testing.equal[i32](t, result, 5)
}

fn test_divide_by_zero(t *testing.T) void {
    result := divide(10, 0) or |err| {
        testing.equal[error](t, err, error.DivideByZero)
        return
    }
    testing.fail(t, "expected error")
}
```

```

```

## Complete TDD Example

**main.yar:**
```yar
package main

fn fizzbuzz(n i32) str {
    if n % 15 == 0 {
        return "FizzBuzz"
    }
    if n % 3 == 0 {
        return "Fizz"
    }
    if n % 5 == 0 {
        return "Buzz"
    }
    return to_str(n)
}

fn main() i32 {
    for i := 1; i <= 20; i += 1 {
        print(fizzbuzz(i) + "\n")
    }
    return 0
}
```

## Dependencies

Git-based, no central registry. Declared in `yar.toml`, pinned in `yar.lock`.

**yar.toml:**
```toml
[package]
name = "myapp"

[dependencies]
http = { git = "https://github.com/user/yar-http.git", tag = "v0.3.1" }
json = { git = "https://github.com/user/yar-json.git", rev = "a1b2c3d" }
local_lib = { path = "../my-local-lib" }
```

- Alias becomes the import path: `import "http"`, `import "http/router"`
- Version: exactly one of `tag`, `rev`, or `branch` per git dependency
- `yar.lock` is an explicit `version = 1` graph: each git package node records
  its exact commit/hash and full alias/git/ref child edges
- Git dependencies in the root manifest and manifests of root path
  dependencies, plus every child edge, must exactly match lock nodes before
  dependency cache or network access
- Duplicate aliases or child edges, missing nodes, cycles, and unreachable
  nodes are rejected; missing/unsupported lock versions require `yar lock`
- `path` dependencies are live and not written to `yar.lock`; only the root
  manifest may declare them. Their manifests may declare git dependencies, but
  may not declare another path dependency
- A selected path alias must exist and does not fall through to a same-named
  stdlib package
- Transitive dependencies supported (each dep may have its own `yar.toml`)
- Conflicts (same alias, different git source, ref kind, or ref value) are
  errors; there is no root override
- Alias visibility is owner-scoped: the entry origin uses root-manifest aliases,
  path origins use their own manifest aliases, and git origins use their lock
  node's child edges
- Lock reachability does not grant import visibility; every source origin must
  directly declare each external alias it imports. Migrate source that imported
  a merely reachable transitive alias by adding it to that source's manifest
- Lock v1/cache formats are unchanged and retain global alias/source uniqueness;
  reusing one alias for different targets in different owner scopes needs lock v2
- A selected locked git dependency is hash-verified before cached source is
  read, then its manifest is checked against the recorded child edges; missing,
  mismatched, or edge-divergent selected entries fail before stdlib fallback,
  while unused or locally shadowed entries do not require a cache
- `yar fetch` verifies existing entries and verifies fresh clones before
  publishing them; cached git trees reject symlinks and special files. A graph
  with no git roots needs no lock file or dependency cache
- Local `path` dependencies remain unhashed filesystem inputs
- `yar update <git-alias>` merges the replacement graph, preserves unrelated
  reachable nodes, refreshes compatible shared nodes, and prunes orphans;
  targeted path updates require `yar lock`

**Commands:**
```bash
yar init                                          # create yar.toml
yar add http https://github.com/u/repo.git --tag=v1.0.0
yar add lib --path=../local-lib                   # local path dep
yar remove http
yar lock                                          # regenerate yar.lock
yar fetch                                         # download to cache
yar update                                        # re-resolve all deps
yar update http                                   # update one git dependency graph
```

Cache: `$HOME/Library/Caches/yar/deps/` on macOS, otherwise
`$HOME/.cache/yar/deps/` (override with `YAR_CACHE`).

## Memory Management

- Runtime-managed heap; the current Rust runtime retains allocations until
  process exit
- No manual free, no GC builtins
- Allocation failure is unrecoverable (not a YAR error)

## Key Restrictions

- No type inference for generics (always explicit: `Box[i32]`)
- No `nil` without pointer context
- No variadics
- No exceptions
- No import aliases
- No `while` or range-for
- Methods are not first-class values
- Raw errorable call expressions must be handled immediately (`?`, `or`, or `return`)
- `void`/`noreturn` cannot be used in parameters, fields, array elements, or channel elements; `[]void` is used for taskgroup results
