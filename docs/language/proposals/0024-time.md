# Proposal: Date and Time (`time` stdlib package)

Status: proposed

## 1. Summary

Add a `time` stdlib package providing wall-clock time, monotonic elapsed
measurement, sleep, date decomposition in UTC and local timezone, date
construction, and RFC 3339 formatting/parsing at both second and nanosecond
precision. Both wall-clock and monotonic time are represented as `i64`
nanoseconds, matching the convention used by Go, Rust, and Zig. Clock, sleep,
and timezone operations are host-backed intrinsics; formatting and parsing are
pure Yar built on top.

## 2. Motivation

Yar programs can read files, run processes, and communicate over the network,
but cannot read the clock, measure elapsed time, sleep, or work with dates.
Even basic tasks like logging with timestamps, benchmarking code, implementing
retry delays, or generating human-readable dates require shelling out to
external tools via `process.run`.

Adding `time` as a host-backed stdlib package follows the same pattern
established by `fs`, `process`, `env`, `stdio`, and `net`. The package fills
the last major gap in Yar's ability to write practical systems programs.

## 3. User-Facing Examples

### Valid examples

```
import "time"

// Print current UTC timestamp
fn main() !i32 {
    ns := time.now()
    s := time.format_rfc3339(ns)?
    print(s + "\n")
    return 0
}
```

```
import "time"

// Measure elapsed time
fn main() i32 {
    start := time.monotonic()
    // ... work ...
    elapsed := time.monotonic() - start
    print("elapsed: " + to_str(elapsed) + " ns\n")
    return 0
}
```

```
import "time"

// Sleep between retries
fn fetch_with_retry(attempts i32) !str {
    for i := 0; i < attempts; i += 1 {
        result := try_fetch() or |err| {
            if i + 1 < attempts {
                time.sleep(time.seconds(1))
            }
            continue
        }
        return result
    }
    return error.MaxRetries
}
```

```
import "time"

// Decompose a timestamp into date components
fn main() !i32 {
    ns := time.now()
    d := time.date(ns)?
    print("year: " + to_str(d.year) + "\n")
    print("month: " + to_str(d.month) + "\n")
    print("day: " + to_str(d.day) + "\n")
    return 0
}
```

```
import "time"

// Construct a timestamp from components
fn main() !i32 {
    ns := time.from_date(2025, 6, 15, 12, 0, 0)?
    print(time.format_rfc3339(ns)? + "\n")
    // Output: 2025-06-15T12:00:00Z
    return 0
}
```

```
import "time"

// Parse an RFC 3339 timestamp
fn main() !i32 {
    ns := time.parse_rfc3339("2025-06-15T12:00:00Z")?
    d := time.date(ns)?
    print("day: " + to_str(d.day) + "\n")
    return 0
}
```

```
import "time"

// Format with nanosecond precision
fn main() !i32 {
    ns := time.now()
    s := time.format_rfc3339_nanos(ns)?
    print(s + "\n")
    // Output: 2025-06-15T12:00:00.123456789Z
    return 0
}
```

```
import "time"

// Work with local timezone
fn main() !i32 {
    ns := time.now()
    d := time.local_date(ns)?
    print("local hour: " + to_str(d.hour) + "\n")
    print("utc offset: " + to_str(d.utc_offset) + " seconds\n")

    // Format local time with offset
    s := time.format_date_rfc3339(d)?
    print(s + "\n")
    // Output: 2025-06-15T08:00:00-04:00  (if EDT)
    return 0
}
```

```
import "time"

// Construct local time and convert to UTC nanos
fn main() !i32 {
    ns := time.from_local_date(2025, 6, 15, 8, 0, 0)?
    utc := time.date(ns)?
    print("utc hour: " + to_str(utc.hour) + "\n")
    // Output: utc hour: 12  (if EDT, offset -4h)
    return 0
}
```

```
import "time"

// Convert to a named IANA timezone
fn main() !i32 {
    ns := time.now()
    d := time.date_in(ns, "Asia/Kolkata")?
    print("IST hour: " + to_str(d.hour) + "\n")
    print("offset: " + to_str(d.utc_offset) + "\n")
    // Output: offset: 19800  (UTC+5:30)
    return 0
}
```

```
import "time"

// Construct a timestamp in a named timezone
fn main() !i32 {
    ns := time.from_date_in(2025, 12, 25, 0, 0, 0, "America/New_York")?
    s := time.format_rfc3339(ns)?
    print(s + "\n")
    // Output: 2025-12-25T05:00:00Z  (EST is UTC-5)
    return 0
}
```

```
import "time"

// Duration helpers make sleep and deadlines readable
fn main() !i32 {
    time.sleep(time.seconds(2))
    time.sleep(time.milliseconds(500))

    deadline := time.now() + time.minutes(30)
    print("deadline: " + time.format_rfc3339(deadline)? + "\n")
    return 0
}
```

### Invalid examples

```
// Cannot mix wall-clock and monotonic nanoseconds
start := time.now()
elapsed := time.monotonic() - start  // wrong: different epoch, meaningless result
```

Wall-clock `now()` returns nanoseconds since Unix epoch. Monotonic
`monotonic()` returns nanoseconds from an arbitrary platform-dependent epoch.
Both are `i64` nanoseconds but not interchangeable. The type system does not
prevent misuse. This is consistent with how `net` uses opaque `i64` for both
listeners and connections.

## 4. Semantics

### Representation

- Wall-clock time: `i64` nanoseconds since 1970-01-01T00:00:00Z (Unix epoch).
  Signed i64 nanoseconds cover approximately 1677 to 2262, matching Go's
  `time.Time.UnixNano()` range. This is the standard representation used by
  Go, Rust (`SystemTime`), Zig (`std.time.nanoTimestamp`), and C
  (`clock_gettime`).
- Monotonic time: `i64` nanoseconds from an unspecified platform-dependent
  epoch. Monotonic time never goes backward and is not affected by clock
  adjustments. Only differences between two `monotonic()` calls are meaningful.

### Functions

Host-backed:

- `now()` returns the current UTC wall-clock time as Unix nanoseconds. Cannot
  fail.
- `monotonic()` returns the current monotonic clock value in nanoseconds.
  Cannot fail. The epoch is unspecified and platform-dependent; only
  differences between two `monotonic()` calls are meaningful.
- `sleep(nanos)` blocks the calling program for at least `nanos` nanoseconds.
  Negative or zero values return immediately. Cannot fail.
- `date(nanos)` decomposes a Unix-nanosecond timestamp into UTC date
  components. Returns a `Date` struct with `utc_offset = 0` and the
  sub-second nanosecond remainder preserved in the `nanosecond` field. Fails
  with `error.InvalidArgument` if the value is outside the representable
  range for the platform's time functions.
- `local_date(nanos)` decomposes a Unix-nanosecond timestamp into local
  timezone date components using the system timezone. Returns a `Date` struct
  with `utc_offset` set to the local UTC offset in seconds at that instant
  (accounts for DST). Fails with `error.InvalidArgument` if the value is
  outside the representable range.
- `date_in(nanos, tz)` decomposes a Unix-nanosecond timestamp into date
  components for the named IANA timezone (e.g., `"America/New_York"`,
  `"Europe/London"`, `"UTC"`). Returns a `Date` struct with `utc_offset` set
  to the timezone's UTC offset at that instant (accounts for DST). Uses the
  system timezone database via C library `TZ` environment variable
  manipulation — safe because Yar is single-threaded. Fails with
  `error.NotFound` if the timezone name is not found in the system database,
  or `error.InvalidArgument` if the timestamp is out of range.
- `from_date(year, month, day, hour, minute, second)` constructs a
  Unix-nanosecond timestamp from UTC date components (nanosecond remainder is
  zero). Normalizes overflow (e.g., month 13 becomes January of the next
  year) following C `mktime` semantics. Fails with `error.InvalidArgument` if
  the resulting time cannot be represented.
- `from_local_date(year, month, day, hour, minute, second)` constructs a
  Unix-nanosecond timestamp from local timezone date components. Uses the
  system timezone to determine the UTC offset at the given local time. Fails
  with `error.InvalidArgument` if the resulting time cannot be represented.
- `from_date_in(year, month, day, hour, minute, second, tz)` constructs a
  Unix-nanosecond timestamp from date components interpreted in the named
  IANA timezone. Uses the system timezone database. Fails with
  `error.NotFound` if the timezone is not found, or
  `error.InvalidArgument` if the result cannot be represented.

Pure Yar — formatting and parsing:

- `format_rfc3339(nanos)` formats a Unix-nanosecond timestamp as an RFC 3339
  string in UTC at second precision (`"2006-01-02T15:04:05Z"`). Implemented
  on top of `date()`. Fails if `date()` fails.
- `format_rfc3339_nanos(nanos)` formats a Unix-nanosecond timestamp as an
  RFC 3339 string in UTC at nanosecond precision
  (`"2006-01-02T15:04:05.123456789Z"`). Trailing zeros in the fractional
  part are preserved (always 9 digits). Implemented on top of `date()`. Fails
  if `date()` fails.
- `format_date_rfc3339(d)` formats a `Date` struct as an RFC 3339 string
  using the `utc_offset` field to produce the correct offset suffix. UTC
  dates produce `"...Z"`, offset dates produce `"...+05:30"` or
  `"...-04:00"`. Second precision. Fails with `error.InvalidArgument` if the
  offset is not a whole-minute value.
- `format_date_rfc3339_nanos(d)` same as `format_date_rfc3339` but includes
  the 9-digit nanosecond fractional part from `d.nanosecond`.
- `parse_rfc3339(s)` parses an RFC 3339 string into Unix nanoseconds.
  Accepts `"YYYY-MM-DDTHH:MM:SSZ"`, `"YYYY-MM-DDTHH:MM:SS±HH:MM"`, and
  optional fractional seconds (`".123"`, `".123456789"`). Second value 60
  (leap second) is accepted and normalized to second 0 of the next minute.
  Returns the corresponding UTC instant. Fails with `error.InvalidFormat` if
  the string does not match.

Pure Yar — duration helpers:

Following Zig's approach of named conversion functions (Yar has no constants),
these make duration arithmetic readable without introducing a new type:

- `seconds(n i64) i64` — returns `n * 1_000_000_000`
- `milliseconds(n i64) i64` — returns `n * 1_000_000`
- `microseconds(n i64) i64` — returns `n * 1_000`
- `minutes(n i64) i64` — returns `n * 60_000_000_000`
- `hours(n i64) i64` — returns `n * 3_600_000_000_000`

Usage: `time.sleep(time.seconds(5))`, `deadline := time.now() + time.hours(1)`.
No `days()` or larger — deliberate omission to avoid DST confusion, following
Go's rationale for excluding `time.Day`.

### Date struct

```
pub struct Date {
    year i32
    month i32         // 1 = January through 12 = December
    day i32           // 1 through 31
    hour i32          // 0 through 23
    minute i32        // 0 through 59
    second i32        // 0 through 59
    nanosecond i32    // 0 through 999_999_999
    weekday i32       // 0 = Sunday through 6 = Saturday
    utc_offset i32    // seconds east of UTC (0 for UTC, -18000 for EST, 19800 for IST)
}
```

The `nanosecond` field preserves sub-second precision from the original `i64`
timestamp. The `utc_offset` field records which timezone the date was
decomposed in: `date()` always sets it to 0, `local_date()` sets it to the
system timezone offset at that instant (which varies with DST).

## 5. Type Rules

- `Date` is a public struct with 9 `i32` fields.
- `now()` and `monotonic()` return `i64`. Not errorable.
- `sleep(nanos i64)` returns `void`. Not errorable.
- `date(nanos i64)` returns `!Date`.
- `local_date(nanos i64)` returns `!Date`.
- `date_in(nanos i64, tz str)` returns `!Date`.
- `from_date(year, month, day, hour, minute, second i32)` returns `!i64` (nanoseconds).
- `from_local_date(year, month, day, hour, minute, second i32)` returns `!i64` (nanoseconds).
- `from_date_in(year, month, day, hour, minute, second i32, tz str)` returns `!i64` (nanoseconds).
- `format_rfc3339(nanos i64)` returns `!str`.
- `format_rfc3339_nanos(nanos i64)` returns `!str`.
- `format_date_rfc3339(d Date)` returns `!str`.
- `format_date_rfc3339_nanos(d Date)` returns `!str`.
- `parse_rfc3339(s str)` returns `!i64` (nanoseconds).
- `seconds(n i64)`, `milliseconds(n i64)`, `microseconds(n i64)`, `minutes(n i64)`, `hours(n i64)` return `i64`. Not errorable.
- All types are existing Yar types. No new type constructors.

## 6. Grammar / Parsing Shape

No language grammar changes. The `time` package is a stdlib package imported
with `import "time"`. All functions are called with qualified names
(`time.now`, `time.date`, etc.).

## 7. Lowering / Implementation Model

### Parser impact

None.

### AST / IR impact

None.

### Checker impact

- `IsHostIntrinsic` extended with 9 host-backed function names: `time.now`,
  `time.monotonic`, `time.sleep`, `time.date`, `time.local_date`,
  `time.date_in`, `time.from_date`, `time.from_local_date`,
  `time.from_date_in`.
- `registerHostErrorNames` extended to register error names:
  `InvalidArgument`, `InvalidFormat`, `NotFound`.
- `InvalidArgument` and `NotFound` are already registered by other packages;
  `InvalidFormat` is new.

### Codegen impact

- 9 new LLVM extern declarations (`@yar_time_*`) in `writeRuntimeDecls`.
- 9 new cases in `genHostIntrinsicCall` following existing patterns.
- `date`, `local_date`, and `date_in` return a struct with 9 fields,
  following the same pattern as `net.listener_addr` returning `net.Addr`
  (alloca struct → call with out-pointer → load).
- `date_in` and `from_date_in` accept a `str` parameter for the timezone
  name, following the same str-passing pattern as `fs.read_file`.
- Error code mapping for `InvalidArgument` (reused), `NotFound` (reused),
  and `InvalidFormat` (new).

### Runtime impact

- ~350 lines of new C code in `runtime_source.txt`.
- POSIX: `clock_gettime(CLOCK_REALTIME)` for `now`,
  `clock_gettime(CLOCK_MONOTONIC)` for `monotonic`, `nanosleep` for `sleep`
  (converts i64 nanos to `struct timespec`), `gmtime_r` for `date` (divides
  nanos to seconds, extracts nanosecond remainder), `localtime_r` for
  `local_date` (same division, plus `tm_gmtoff` for UTC offset), `timegm` for
  `from_date` (multiplies seconds to nanos), `mktime` for `from_local_date`.
- Named timezone support (`date_in`, `from_date_in`): saves the current `TZ`
  environment variable, sets `TZ` to the requested IANA name, calls `tzset()`
  then `localtime_r` / `mktime`, restores the original `TZ`, and calls
  `tzset()` again. Safe because Yar is single-threaded. Validates that the
  timezone was applied by checking `tm_zone` (POSIX) or verifying the offset
  changed for a known non-UTC timezone. Returns `error.NotFound` status if
  the timezone name does not resolve to a valid entry in the system's
  `/usr/share/zoneinfo` database.
- Windows: `GetSystemTimeAsFileTime` for `now` (converts 100ns FILETIME ticks
  to Unix nanos), `QueryPerformanceCounter`/`QueryPerformanceFrequency` for
  `monotonic`, `Sleep` for `sleep` (converts nanos to milliseconds, rounding
  up), `gmtime_s` for `date`, `localtime_s` for `local_date` (with
  `_get_timezone`/`_get_daylight` for UTC offset), `_mkgmtime` for
  `from_date`, `mktime` for `from_local_date`. Named timezone on Windows
  uses `_putenv_s("TZ", ...)` plus `_tzset()` with the same save/restore
  pattern.
- No new library dependencies on any platform.

### Pure Yar functions

- Format, parse, and duration helpers are implemented in `time.yar` as
  ordinary Yar code, importing `strings` and `conv` as needed.
- `format_rfc3339` calls `time.date()` then formats each field with
  zero-padding, appends `"Z"`.
- `format_rfc3339_nanos` is the same but appends `".NNNNNNNNNZ"` using the
  `nanosecond` field from `Date`.
- `format_date_rfc3339` and `format_date_rfc3339_nanos` use the `utc_offset`
  field to produce `"Z"` (offset 0) or `"±HH:MM"` suffix.
- `parse_rfc3339` validates the string shape, extracts substrings, parses
  integers with `strings.parse_i64`, handles optional fractional seconds and
  `"Z"` or `"±HH:MM"` offset, adjusts for the parsed offset, and calls
  `time.from_date()`. Second value 60 (leap second) is normalized to second 0
  of the next minute.
- Duration helpers (`seconds`, `milliseconds`, `microseconds`, `minutes`,
  `hours`) are single-line multiplication functions.

## 8. Interactions

### Errors

- `error.InvalidArgument` is reused from existing packages (`process`, `env`,
  `net`). No conflict.
- `error.NotFound` is reused (used by `date_in` / `from_date_in` when the
  timezone name is not found in the system database).
- `error.InvalidFormat` is new. It joins the program-wide error code table
  when any `time` function that produces it is called.
- All error handling works through standard `?` and `or |err| { ... }`.

### Structs

`time.Date` is a regular public struct with 9 fields. Field access, literals,
and passing work as expected. The `nanosecond`, `weekday`, and `utc_offset`
fields are computed by `date()` / `local_date()` / `date_in()` — `from_date`,
`from_local_date`, and `from_date_in` do not accept them as parameters,
keeping the construction API focused on the 6 essential components.

### Control flow

`sleep` blocks the program. In a single-threaded program this pauses all
execution. No new control flow mechanisms.

### Builtins

No new builtins. `to_str` works on `Date` fields (`i32`).

### net package

The `net` package uses millisecond timeouts (`set_read_deadline`,
`set_write_deadline`). Converting `time` nanoseconds to `net` milliseconds
requires dividing by 1,000,000. This is explicit and straightforward.

## 9. Alternatives Considered

### 1. Millisecond wall-clock time

Would align with `net` timeout values and JavaScript conventions. Rejected
because nanoseconds are the standard precision used by Go, Rust, Zig, and C.
Milliseconds would lose precision unnecessarily and be the odd choice out.
Converting to milliseconds for `net` deadlines is a single division.

### 2. Struct-based time value (`Time { sec i64, nsec i32 }`)

Would avoid the i64 nanosecond range limitation (~1677–2262). Rejected because
the range is practical for real-world programs, a single `i64` enables simple
arithmetic (subtraction for duration, comparison with `<`/`>`), and a struct
type can be introduced later as a pure Yar wrapper if the range proves
insufficient.

### 3. Embedded timezone database (Go approach)

Go embeds ~450KB of IANA tzdata into every binary via `time/tzdata`. This
guarantees timezone support on platforms without a system tzdata installation.
Rejected because it inflates every Yar binary even when timezone support is
unused, and the system tzdata is available on all major platforms (Linux
distros, macOS, Windows). The C library `TZ` + `tzset()` approach uses the
system database at zero binary cost. If a platform lacks tzdata, `date_in` /
`from_date_in` return `error.NotFound`, which is explicit and handleable.

### 4. Duration as a named type or struct (Go/Rust approach)

Go defines `type Duration int64` with methods. Rust defines `Duration { secs,
nanos }` with extensive arithmetic. Both add type safety but require operator
overloading or method chains for basic arithmetic. Rejected because Yar has
neither operator overloading nor type aliases, so a Duration type would require
explicit constructors and method calls for every operation. Zig's approach —
raw `i64` nanoseconds with named conversion functions — fits Yar's type system
naturally: `time.sleep(time.seconds(5))` reads well and `time.now() +
time.hours(1)` works with plain `+`. No `days()` or larger, following Go's
deliberate omission to avoid DST confusion.

### 5. Timer / ticker (Go approach)

Go provides `time.NewTimer` and `time.NewTicker` backed by goroutines and
channels. Rust provides `tokio::time::interval` in async contexts. Zig has no
timer facility. Rejected because Yar is single-threaded with no channels,
goroutines, or async primitives. Timers and tickers are fundamentally
concurrency constructs. Users loop with `time.sleep` for periodic work. If
Yar adds concurrency primitives in the future, timer support can be revisited.

### 6. Custom format strings

Would enable arbitrary date formatting like `"Mon Jan 2 15:04:05 2006"`.
Rejected because format-string parsing is complex, error-prone, and a large
implementation surface. RFC 3339 is the standard for machine-readable
timestamps. Additional format functions can be added to the package later
without breaking changes.

### 7. Package name `datetime` or `clock`

`datetime` is unnecessarily long by Yar stdlib conventions (`fs`, `net`,
`env`). `clock` is too narrow — implies only reading time, not dates or
formatting. `time` is the standard name used by Go, Python, and C.

## 10. Complexity Cost

- **Language surface**: zero (no grammar changes)
- **Parser complexity**: zero
- **Checker complexity**: minimal (1 switch case, error name registration)
- **Codegen complexity**: moderate (9 new intrinsic cases, follows existing
  patterns exactly, struct return reuses `net.Addr` pattern)
- **Runtime complexity**: moderate (~350 lines of C, standard POSIX/Win32
  APIs, TZ save/restore for named timezones)
- **Pure Yar complexity**: moderate (~180 lines for format/parse with offset
  handling, nanosecond formatting, leap second normalization, and duration
  helpers)
- **Diagnostics complexity**: zero
- **Test burden**: 1 integration test
- **Documentation burden**: moderate (stdlib docs, error model, context docs)

## 11. Why Now?

Time operations are the primary remaining gap after networking was added.
Practical programs need timestamps for logging, elapsed measurement for
performance work, sleep for retry loops and rate limiting, and date formatting
for human-readable output. The host-intrinsic pattern is well-established and
this package follows it exactly. The API surface is complete (9 host functions,
10 pure Yar functions, 1 struct) and introduces no new compiler infrastructure.

## 12. Resolved Design Decisions

Each decision was informed by surveying Go, Rust, and Zig implementations.

### Named timezones — included, using system tzdata

**Go** embeds ~450KB of IANA tzdata via `time/tzdata`. **Rust** has no stdlib
support; `chrono-tz` embeds the IANA database at compile time. **Zig** has no
stdlib support; third-party libraries embed or load tzdata at runtime.

**Decision**: include `date_in` and `from_date_in` using the system's timezone
database via C library `TZ` environment variable manipulation. Yar is
single-threaded, so the non-thread-safe `setenv`/`tzset`/`localtime_r` pattern
is safe. This provides zero binary size overhead (unlike Go's embedded
approach) while supporting all IANA timezone names on platforms with tzdata
installed. Platforms without tzdata get a clean `error.NotFound`.

### Duration — Zig-style conversion functions, no new type

**Go** defines `type Duration int64` with constants (`time.Second`) and
methods (`.Seconds()`). **Rust** defines `Duration { secs: u64, nanos: u32 }`
with constructors (`from_secs`, `from_millis`). **Zig** uses raw `u64` with
conversion constants (`ns_per_s`, `ns_per_ms`).

**Decision**: follow Zig's approach adapted for Yar — pure functions
(`seconds`, `milliseconds`, `microseconds`, `minutes`, `hours`) that multiply
to nanoseconds. Yar has no constants, no type aliases, and no operator
overloading, so Go's `Duration` and Rust's `Duration` struct don't translate
well. Functions work naturally: `time.sleep(time.seconds(5))`. No `days()` or
larger — following Go's deliberate omission to avoid DST confusion (a "day" is
not always 24 hours).

### Timer / ticker — not included

**Go** provides `time.NewTimer` and `time.NewTicker` using goroutines and
channels. **Rust** provides `tokio::time::interval` for async contexts.
**Zig** has no timer facility.

**Decision**: not included. Timers and tickers are concurrency constructs that
require channels, goroutines, or async primitives — none of which Yar has.
Users loop with `time.sleep` for periodic work. Matches Zig's approach.

### Leap seconds — accept and normalize (second=60)

**Go** rejects second=60 entirely in `time.Parse` (returns error). **Rust**'s
`chrono` encodes leap seconds via subsecond overflow (1000ms at second=59).
**Zig** has no leap second handling.

**Decision**: `parse_rfc3339` accepts second=60 and normalizes it to second=0
of the next minute, preserving the UTC instant. This is pragmatic — RFC 3339
explicitly allows `23:59:60Z` and real-world timestamps from NTP-aware systems
produce them. Rejecting (Go) loses data; subsecond encoding (Rust/chrono) is
clever but complex. Normalizing is simple, correct, and round-trips through
`format` → `parse` predictably (the normalized form). `date()` /
`local_date()` / `date_in()` never produce second=60 because `gmtime_r` and
`localtime_r` do not.
