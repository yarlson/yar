# Proposal: Time Values and UTC (`time` stdlib package)

Status: proposed

This revision supersedes the earlier raw-`i64` and timezone design. It is not
implemented and does not describe current standard-library behavior.

## 1. Decision Summary

The first `time` package must keep three temporal domains nominally distinct:

- `Timestamp` is a wall-clock coordinate on the Unix UTC timeline.
- `Instant` is a process-local monotonic clock reading.
- `Duration` is a signed span.

The first delivery includes wall and monotonic clocks, checked duration and
timestamp operations, blocking sleep, strict UTC calendar conversion, and RFC
3339 parsing and formatting. Local time, named timezones, timezone-database
ownership, timers, tickers, and cancellation-aware sleep are separate designs.

No operation accepts interchangeable raw clock values. In particular,
subtracting an `Instant` from a `Timestamp` is a type error.

## 2. Types

```yar
pub struct Timestamp {
    unix_nanoseconds i64
}

pub struct Instant {
    monotonic_nanoseconds i64
}

pub struct Duration {
    nanoseconds i64
}

pub struct Date {
    year i32
    month i32
    day i32
    hour i32
    minute i32
    second i32
    nanosecond i32
    weekday i32
}
```

These wrappers provide nominal type separation with the current language.
Yar does not yet support private fields on public value structs, so callers can
construct or inspect their representation. This is an acknowledged provenance
loophole, not permission to mix domains. Public functions validate every
invariant that can be violated at their boundary. A future representation-
privacy feature may hide the fields without changing the function signatures.

### `Timestamp`

`unix_nanoseconds` is signed nanoseconds from
`1970-01-01T00:00:00Z`. Negative values represent pre-epoch timestamps. The
`i64` range covers approximately years 1677 through 2262. Timestamp resolution
is one nanosecond; host clock precision, accuracy, and resolution are not
promised to be one nanosecond. Wall time may move backward or forward after a
clock adjustment.

### `Instant`

`monotonic_nanoseconds` is measured from one process-local runtime origin.
Only values created in the same process execution are meaningful. They must
not be persisted, sent to another process, or interpreted as wall time.
Successive readings are nondecreasing. Whether elapsed time includes system
suspend is intentionally unspecified because supported hosts differ.

The public representation means a caller can forge an `Instant`; the runtime
cannot prove its provenance. The API still prevents accidental cross-domain
use, and reversed monotonic differences are rejected.

### `Duration`

`Duration` is signed because wall-clock differences may be negative. Zero is a
valid duration. Scaled constructors and arithmetic never silently wrap.
`sleep` accepts only non-negative durations.

### `Date`

`Date` is a UTC Gregorian decomposition. Fields use these ranges:

- `month`: 1 through 12
- `day`: valid day for the selected year and month
- `hour`: 0 through 23
- `minute`: 0 through 59
- `second`: 0 through 59
- `nanosecond`: 0 through 999,999,999
- `weekday`: 0 for Sunday through 6 for Saturday

Leap-second value 60 is rejected. The package uses the Unix/POSIX timescale and
does not claim to model leap seconds.

## 3. API

### Clocks and sleep

```yar
pub fn now() !Timestamp
pub fn instant() Instant
pub fn sleep(duration Duration) !void
```

- `now` reads the host wall clock and converts it to signed Unix nanoseconds.
  It returns `error.Overflow` if the value cannot fit the `Timestamp` range.
- `instant` reads a monotonic clock relative to the process runtime origin.
  Exhausting its range is a deterministic runtime failure rather than wrap.
- `sleep` returns immediately for zero, rejects a negative duration with
  `error.InvalidArgument`, and otherwise blocks the calling native task thread
  for at least the requested span subject to host scheduling.

Sleep is not a deadline, timer, cancellation point, or scheduler primitive.
Inside a taskgroup it does not block sibling native task threads, but the
taskgroup still waits for the sleeping task.

### Duration construction and inspection

```yar
pub fn nanoseconds(value i64) Duration
pub fn microseconds(value i64) !Duration
pub fn milliseconds(value i64) !Duration
pub fn seconds(value i64) !Duration
pub fn minutes(value i64) !Duration
pub fn hours(value i64) !Duration
pub fn duration_nanoseconds(value Duration) i64
```

`nanoseconds` cannot overflow. Every scaled constructor performs checked
multiplication and returns `error.Overflow` instead of relying on Yar's normal
wrapping integer arithmetic. Negative inputs remain negative durations.
There is no `days` helper because calendar days are not fixed elapsed spans in
timezone-aware work.

### Timestamp and instant operations

```yar
pub fn timestamp_from_unix_nanoseconds(value i64) Timestamp
pub fn timestamp_unix_nanoseconds(value Timestamp) i64

pub fn timestamp_add(value Timestamp, span Duration) !Timestamp
pub fn timestamp_subtract(value Timestamp, span Duration) !Timestamp
pub fn timestamp_difference(left Timestamp, right Timestamp) !Duration
pub fn timestamp_before(left Timestamp, right Timestamp) bool
pub fn timestamp_equal(left Timestamp, right Timestamp) bool

pub fn instant_difference(later Instant, earlier Instant) !Duration
pub fn instant_before(left Instant, right Instant) bool
pub fn instant_equal(left Instant, right Instant) bool
```

Explicit functions replace operators, which Yar does not define for structs.
Timestamp arithmetic and differences return `error.Overflow` if the `i64`
result cannot be represented. `timestamp_difference(left, right)` computes
`left - right`, so its duration may be negative.
`instant_difference` returns `error.InvalidArgument` when `later` precedes
`earlier`, and `error.Overflow` if subtraction cannot be represented.

There is deliberately no conversion between `Timestamp` and `Instant`.

### UTC calendar and RFC 3339

```yar
pub fn utc_date(value Timestamp) Date
pub fn from_utc(
    year i32,
    month i32,
    day i32,
    hour i32,
    minute i32,
    second i32,
    nanosecond i32,
) !Timestamp
pub fn format_rfc3339(value Timestamp) str
pub fn parse_rfc3339(value str) !Timestamp
```

`utc_date` uses deterministic proleptic-Gregorian arithmetic over the complete
`Timestamp` range. `from_utc` validates every input strictly and computes the
weekday rather than accepting it as redundant input. It does not normalize
month 13, an out-of-range day, `24:00:00`, or a leap second. Invalid calendar
fields return `error.InvalidArgument`. Individually valid fields outside the
representable `Timestamp` interval return `error.Overflow`; only part of the
boundary years 1677 and 2262 is representable.

`format_rfc3339` always emits UTC with `Z`. It omits the fractional component
when zero and otherwise emits up to nine digits with trailing zeros removed.
The supported timestamp range always has a four-digit year.

`parse_rfc3339` consumes the entire input and requires fixed-width
`YYYY-MM-DDTHH:MM:SS`, followed by an optional dot and one-to-nine fractional
digits, then either `Z` or a numeric `+HH:MM` / `-HH:MM` offset. Separators are
uppercase `T` and `Z` only. Calendar validation rejects `24:00:00` and leap
seconds. Numeric offset hours are `00` through `23`, minutes are `00` through
`59`, and `+24:00` and larger values are rejected. `-00:00` is rejected because
RFC 3339 uses it to mean an unknown local offset rather than UTC; `+00:00` is
accepted. Malformed text returns `error.InvalidFormat`. A syntactically valid
value whose offset adjustment exceeds the Timestamp range returns
`error.Overflow`.

### Errors

| Function | Source-level errors |
| --- | --- |
| `now` | `Overflow` |
| scaled duration constructors | `Overflow` |
| `sleep` | `InvalidArgument` for a negative duration |
| timestamp add, subtract, and difference | `Overflow` |
| `instant_difference` | `InvalidArgument` for reversed values; `Overflow` |
| `from_utc` | `InvalidArgument` for invalid fields; `Overflow` |
| `parse_rfc3339` | `InvalidFormat`; `Overflow` |

## 4. Type Rules

No grammar changes or temporal operators are introduced. The package is
imported as `std/time`, and ordinary nominal struct typing rejects domain
mixing:

```yar
start := time.instant()
stamp := time.now()?

time.sleep(stamp)                       // invalid: Timestamp is not Duration
time.instant_difference(start, stamp)  // invalid: Timestamp is not Instant
stamp + time.seconds(1)?                // invalid: no struct arithmetic
```

Callers must opt into representation access to forge or reconstruct a value.
That limitation is visible and testable; it is not hidden behind claims of
full opacity.

## 5. Implementation Boundary

Only three operations require host support. Their exact exported C ABI is:

```text
i32 yar_time_now_unix_nanoseconds(i64 *out)
i64 yar_time_instant_nanoseconds(void)
i32 yar_time_sleep_nanoseconds(i64 nanoseconds)
```

`yar_time_now_unix_nanoseconds` validates `out` before reading the clock,
returns status 0 on success, and maps a conversion/range failure to
`error.Overflow`. `yar_time_sleep_nanoseconds` returns status 0 on success and
maps negative input to `error.InvalidArgument`. The instant function returns a
direct scalar and terminates with
`runtime failure: monotonic clock range exhausted` if its process-relative
range is exhausted.

These are private host intrinsics used by pure-Yar public wrappers. Duration
construction, checked arithmetic, comparisons, UTC calendar conversion, and
RFC 3339 text handling are deterministic package code.

The runtime implementation should use Rust's supported `SystemTime`,
`Instant`, and `thread::sleep` abstractions rather than separate handwritten
POSIX and Win32 clock code. A thread-safe process origin is initialized before
sampling the current monotonic value. Conversion from Rust's unsigned duration
types to `i64` is checked.

The runtime C ABI uses only direct scalars or pointers to scalar output storage
for these private primitives. No Rust or Yar aggregate is passed or returned by
value. Adding the symbols changes runtime ABI 3 to ABI 4 across the CLI contract
and all five checked-in target manifests. Compiler compatibility metadata does
not change because the bundle's ABI field already makes older bundles fail
validation before linking.

## 6. Concurrency and Operational Semantics

- Clock calls and UTC conversion mutate no process-global state and are safe
  from concurrent tasks.
- The package never changes `TZ`, locale, or another process environment value.
- Sleep consumes the calling task's native OS thread. It has no cancellation
  or wakeup handle.
- A positive sleep promises a lower bound, not an exact wake time or upper
  latency bound.
- Host clock resolution and accuracy are platform-dependent. Nanoseconds are
  the representation unit only.
- Wall-clock comparisons and differences remain vulnerable to ordinary clock
  adjustments; elapsed measurement must use `Instant`.

## 7. Deferred Scope

The first package does not include:

- local timezone conversion
- named IANA or Windows timezone conversion
- timezone database loading, caching, or version ownership
- daylight-saving gap/fold resolution policies
- process-global `TZ` mutation
- custom or locale-sensitive formatting
- HTTP-date formatting
- leap-second modeling
- timers, tickers, interval streams, or channels
- cancellation-aware sleep or deadlines

A future timezone proposal must define immutable explicit location values,
tzdata ownership and versioning, Windows/IANA mapping, and policies for
ambiguous and nonexistent local times. It must not use save/set/restore of the
process environment.

## 8. Rejected Prior Design

The superseded proposal used `i64` for wall time, monotonic readings, and
durations. Its own examples admitted that meaningless cross-domain arithmetic
compiled. Named conversions mutated `TZ` and claimed this was safe because Yar
was single-threaded, which is false with native-thread taskgroups. It also
treated nanosecond representation as host precision, allowed scaled duration
helpers to wrap, normalized invalid dates through C library behavior, and
promised portable IANA zones through incompatible platform facilities.

Those choices are rejected. Raw representation access remains temporarily
possible because of public struct fields, but the API and type checker no
longer make domain mixing the default path.

## 9. Verification Plan

Implementation is not complete until tests cover:

- compile-time rejection of every cross-domain argument and raw struct
  arithmetic
- duration constructor boundaries at the exact multiplication limits
- checked timestamp addition, subtraction, and difference overflow
- monotonic nondecrease and reversed-instant rejection
- concurrent first-use of the process monotonic origin
- zero, negative, and positive sleep behavior with a generous watchdog
- a sleeping task not blocking a sibling native task
- UTC epoch, pre-epoch dates, leap years, month lengths, and range endpoints
- strict rejection of normalized or malformed UTC construction inputs
- RFC 3339 `Z`, numeric offsets, fractional widths, malformed inputs, leap
  seconds, and range overflow
- Linux, macOS, and Windows native clock/sleep behavior
- all five release target bundles compiling and linking the new ABI
- deterministic null-output failure for `yar_time_now_unix_nanoseconds` and
  deterministic host-conversion failures at the runtime edge

Tests must inject or bracket wall-clock readings instead of asserting an exact
current time. Sleep tests assert that execution is not intentionally early and
use a loose upper watchdog; they do not assume scheduler precision.
