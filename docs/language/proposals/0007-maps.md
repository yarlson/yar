# Proposal: Maps

Status: proposed

## 1. Summary

Add a built-in associative container type:

- `map[K]V`
- map literals
- map element assignment with `m[key] = value`
- map lookup in expression position with explicit missing-key handling
- `has`, `delete`, and `len` support

## 2. Motivation

The current language has fixed arrays and an exploratory slices proposal, but no
associative lookup structure.

Frontend self-hosting creates direct need for:

- function and struct symbol tables
- local-scope maps
- duplicate-detection sets
- visited sets for cycle checks
- error-name to code tables

Without maps, the checker and parser can only be modeled with awkward linear
searches or hand-built container libraries that themselves need map-like
capability.

## 3. User-Facing Examples

### Valid examples

```yar
fn main() !i32 {
    counts := map[str]i32{"main": 1}
    counts["check"] = 2

    if has(counts, "main") {
        x := counts["main"]?
        return x
    }

    return 0
}
```

### Invalid examples

```yar
bad := map[[4]i32]i32{}
```

Invalid because array keys are not part of the minimal supported key set.

```yar
if counts["main"] {
    return 0
}
```

Invalid because map lookup is not a plain `bool`; it is a value that may be
missing and must be handled explicitly.

```yar
counts[missing()?] = 1
```

Invalid because raw errorable values remain disallowed in indexing expressions.

## 4. Semantics

`map[K]V` stores zero or more key/value pairs.

The first version is intentionally minimal:

- supported key types are `bool`, `i32`, `i64`, and `str`
- `m[key] = value` inserts or replaces the entry for `key`
- `m[key]` in expression position performs a lookup and returns `!V`
- a missing key yields a fixed builtin error `error.MissingKey`
- `has(m, key)` reports whether the key exists
- `delete(m, key)` removes the key when present
- `len(m)` returns the current entry count as `i32`

This keeps missing-key behavior explicit and compatible with YAR's existing
error handling model.

The first version does not include map iteration order guarantees, set syntax,
or a separate iterator protocol.

## 5. Type Rules

- `map[K]V` is well-typed when `K` is one of the supported key types and `V` is
  a first-class storable value type
- `m[key] = value` requires `m` to be `map[K]V`, `key` to be `K`, and `value` to
  be `V`
- `m[key]` in expression position has type `!V`
- `has(m, key)` returns `bool`
- `delete(m, key)` returns `void`
- `len(m)` returns `i32`

## 6. Grammar / Parsing Shape

- extend type parsing to support `map[K]V`
- add map literal syntax `map[K]V{key1: value1, key2: value2}`
- reuse existing indexing and assignment syntax for `m[key]` and `m[key] = ...`

Examples:

- `map[str]i32{}`
- `map[str]bool{"seen": true}`
- `symbols[name] = sig`
- `sig := symbols[name]?`

## 7. Lowering / Implementation Model

- parser: add map types and map literals
- AST / IR: add map type references and map literal nodes
- checker: validate key-type restrictions, lookup typing, and builtin map
  operations
- codegen: lower map values to runtime-owned handles or pointers
- runtime: add map allocation, hashing, lookup, insert, delete, and length

Map lookup returning `!V` means no special tuple or optional-value feature is
required. It uses the existing error-handling path instead.

## 8. Interactions

- errors: map lookup composes directly with `?` and `or |err| { ... }`
- structs: structs can be map values
- arrays: arrays can be map values, but not initial key types in the minimal
  version
- control flow: `has` gives a direct boolean check for branch conditions
- returns: lookup results can propagate like other `!T` values
- builtins: extends `len` and adds fixed map builtins
- future modules/imports: maps are central to symbol resolution and package
  indexing
- future richer type features: enums would be strong future key-type candidates

## 9. Alternatives Considered

### Wait for user-defined libraries only

Rejected because building a usable map library without any associative primitive
creates a chicken-and-egg problem.

### Map lookup returns zero value on miss

Rejected because it hides an important distinction and makes absent keys easier
to misuse.

### Separate set type first

Rejected because maps can cover the core use cases while keeping the initial
surface smaller.

## 10. Complexity Cost

- language surface: medium
- parser complexity: medium
- checker complexity: medium
- lowering/codegen complexity: high
- runtime complexity: high
- diagnostics complexity: medium
- test burden: high
- documentation burden: medium

## 11. Why Now?

Maps are not just general convenience. They are a direct blocker for self-hosted
semantic analysis and name resolution.

## 12. Open Questions

- Should the first version guarantee any map literal duplicate-key behavior?
- Should `error.MissingKey` be the only lookup failure, or should some runtime
  failures surface separately?
- Should future versions permit enum keys immediately when enums arrive?

## 13. Decision

Pending.

This proposal favors explicit missing-key handling and a deliberately small key
set for the first version.
